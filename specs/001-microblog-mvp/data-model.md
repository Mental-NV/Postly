# Data Model: Postly Microblog MVP

## Overview

The MVP data model stays intentionally small and centered on the five user-facing
capabilities in the approved spec: accounts, sessions, posts, follows, and likes.
SQLite is the source of truth, EF Core manages schema/migrations, and no
denormalized read models or repository wrappers are introduced up front.

## Entities

### UserAccount

| Field | Type | Rules / Notes |
|-------|------|---------------|
| `Id` | `long` | Primary key |
| `Username` | `string` | Stored in trimmed display form; 3-20 chars; letters, digits, underscores |
| `NormalizedUsername` | `string` | Uppercased/invariant form for case-insensitive uniqueness |
| `DisplayName` | `string` | Required; trimmed; 1-50 chars |
| `Bio` | `string?` | Optional; max 160 chars |
| `PasswordHash` | `string` | Generated by ASP.NET password hasher |
| `CreatedAtUtc` | `DateTimeOffset` | Immutable creation timestamp |

**Derived/read-model values**

- Default avatar initials derive from `DisplayName`.
- Default avatar visual style derives deterministically from `NormalizedUsername`.
- `FollowerCount` and `FollowingCount` are projected from `Follow`.

### Session

| Field | Type | Rules / Notes |
|-------|------|---------------|
| `Id` | `Guid` | Primary key for session row |
| `UserAccountId` | `long` | FK to `UserAccount` |
| `TokenHash` | `string` | Hash of opaque cookie token; raw token never persisted |
| `CreatedAtUtc` | `DateTimeOffset` | Session issue time |
| `ExpiresAtUtc` | `DateTimeOffset` | Hard expiry used for sign-in bootstrap and auth checks |
| `RevokedAtUtc` | `DateTimeOffset?` | Set on sign-out or forced invalidation |
| `LastSeenAtUtc` | `DateTimeOffset?` | Optional rolling activity timestamp for diagnostics |

### Post

| Field | Type | Rules / Notes |
|-------|------|---------------|
| `Id` | `long` | Primary key; used for direct post URLs |
| `AuthorId` | `long` | FK to `UserAccount` |
| `Body` | `string` | Required; 1-280 chars |
| `CreatedAtUtc` | `DateTimeOffset` | Original publish time; timeline order uses this |
| `EditedAtUtc` | `DateTimeOffset?` | Null until first successful edit |

**Derived/read-model values**

- `IsEdited` is `EditedAtUtc != null`.
- `LikeCount` is projected from `Like`.
- `LikedByViewer`, `CanEdit`, and `CanDelete` are request-context booleans,
  not stored columns.

### Follow

| Field | Type | Rules / Notes |
|-------|------|---------------|
| `FollowerId` | `long` | FK to `UserAccount` |
| `FollowedId` | `long` | FK to `UserAccount` |
| `CreatedAtUtc` | `DateTimeOffset` | Relationship creation time |

**Rules**

- Composite key: (`FollowerId`, `FollowedId`)
- `FollowerId != FollowedId`
- Represents a directional relationship only

### Like

| Field | Type | Rules / Notes |
|-------|------|---------------|
| `UserAccountId` | `long` | FK to `UserAccount` |
| `PostId` | `long` | FK to `Post` |
| `CreatedAtUtc` | `DateTimeOffset` | When the like was created |

**Rules**

- Composite key: (`UserAccountId`, `PostId`)
- Used for aggregate like count and current-user liked state only

## Relationships

- `UserAccount 1 -> many Session`
- `UserAccount 1 -> many Post`
- `UserAccount many -> many UserAccount` through `Follow`
- `UserAccount many -> many Post` through `Like`

## Validation Rules

### UserAccount

- `Username` is trimmed before validation and persistence.
- `NormalizedUsername` must be unique.
- `DisplayName` is trimmed before validation and cannot be empty after trimming.
- `Bio` can be null/empty but cannot exceed 160 characters.
- `PasswordHash` is required; raw passwords are never stored.

### Session

- A session is valid only when `RevokedAtUtc` is null and `ExpiresAtUtc` is in
  the future.
- Session tokens are compared by hash.

### Post

- `Body` length must stay between 1 and 280 characters on create and edit.
- Only the author can mutate a post.
- Editing updates `EditedAtUtc` but does not change `CreatedAtUtc`.
- Deletion is a hard delete for MVP simplicity; stale reads become not found.

### Follow

- Self-follow is invalid.
- Duplicate follows are prevented by the composite key.

### Like

- Duplicate likes are prevented by the composite key.
- Like removal on a non-existent like is treated as a no-op at the persistence
  layer and surfaced through a consistent API outcome.

## Indexing Strategy

- Unique index on `UserAccount.NormalizedUsername`
- Index on `Post.AuthorId, Post.CreatedAtUtc DESC, Post.Id DESC`
- Index on `Post.CreatedAtUtc DESC, Post.Id DESC` for timeline reads
- Unique composite index on `Follow(FollowerId, FollowedId)`
- Supporting indexes on `Follow(FollowedId)` for follower counts and
  `Follow(FollowerId)` for following lists
- Unique composite index on `Like(UserAccountId, PostId)`
- Supporting index on `Like(PostId)` for aggregate like counts
- Unique index on `Session.TokenHash`
- Index on `Session.UserAccountId, Session.ExpiresAtUtc`

## State Transitions

### Session Lifecycle

`Active -> Revoked`
`Active -> Expired`

- `Active`: issued and not expired/revoked
- `Revoked`: explicit sign-out or invalidation
- `Expired`: time-based invalidation

### Post Lifecycle

`Published -> Edited -> Edited -> ...`
`Published/Edited -> Deleted`

- Draft state is UI-only and is not persisted in the MVP.
- Deleted posts are removed from visible collections and direct reads return
  not-available/not-found behavior.

## Read Models

### TimelinePage

- List of newest-first `PostSummary` items
- Opaque `NextCursor` for pagination
- Includes current-user-specific flags (`LikedByViewer`, `CanEdit`, `CanDelete`)

### ProfileView

- Profile identity (`Username`, `DisplayName`, `Bio`, derived default avatar)
- `FollowerCount`, `FollowingCount`
- Viewer context (`IsSelf`, `IsFollowedByViewer`, `CanFollow`)
- Paginated post list using the same `PostSummary` contract as timeline/direct
  post reads

### DirectPostView

- Single `PostSummary`
- Not-available state when the post was deleted or cannot be resolved after
  sign-in

## Schema Evolution Notes

- Every schema change must land through an EF Core migration checked into the
  repository.
- Roll-forward is the default migration strategy for the MVP; rollback notes
  must exist for destructive changes.
- Avoid denormalized counters or shadow history tables until query profiles or
  product requirements justify them.
